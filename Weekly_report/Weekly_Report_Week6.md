# 个性化旅游推荐系统 - 第六周工作周报

成员：陈逸 程小路

## 1. 第六周（2026-04-06 至 2026-04-12）：设计冻结后的首轮开发迭代，设计方案讲解

### 1.1 工作目标

本周重点为：

- 按教学节点要求，交付并讲解《设计方案讲解材料》，对齐《项目计划书》中「冻结技术方案与模块边界」目标，收束总体架构、模块边界、接口契约、数据模型及 My* 结构与课程约束说明。
- 在后端持续推进自定义数据结构（MyList / MyMap / MySet / MyPriorityQueue）与核心服务的结合，并完成基础自测与编译验证。
- 保证鉴权、推荐等基础接口在 dev-seed / 放行模式下可稳定调用，接口响应满足统一的 `code` / `data` / `message` 约定。
- 前端巩固登录态与主链路页面框架，完成与路线规划、地图展示相关的高频联调项。
- 管理端与 OSM 生成链路在真实数据场景下可复用、可回归，降低后续迭代返工成本。

### 1.2 具体完成工作

#### 1.2.1 设计方案讲解（教学节点）

- 依据《项目计划书》教学节点对齐表第 5 周要求，补充并完成**设计方案讲解**环节。
- 讲解材料覆盖：前后端分离架构与模块划分、核心接口契约与统一响应结构（`code` / `data` / `message`）、内存预加载与检索约束（不在 SQL 层完成检索计算）、自定义数据结构（MyList / MyMap / MySet / MyPriorityQueue）在算法模块中的定位，以及路线规划 / 推荐 / 日记等业务边界与暂缓项。
- 结合《技术设计文档》与需求文档中的 FR 映射进行现场说明，收集疑问与反馈，并将结论同步到后续开发与交接记录，作为技术方案冻结的验收佐证之一。

#### 1.2.2 路线规划与地图可视化联调

- 后端 `GET /api/route/map-data` 增强：在 `nodes` / `edges` 之外补充 `nodeDetails`，返回节点关联建筑信息（名称、类型、位置、经纬度、`areaId` 等），支撑前端以业务语义选择起终点。
- 前端路线规划页改造：起点 / 终点由手工输入节点 ID 调整为「建筑下拉选择」；图节点标签展示为建筑名称；切换景区时自动刷新地图并重置起止点，避免跨景区误选。
- 路线图渲染策略调整：采用固定布局结合建筑经纬度映射，使拓扑相对位置更贴近真实地理关系；关闭节点拖拽，补充缺失坐标时的兜底布局，保证弱数据下仍可展示。
- 可读性优化：节点标签避让与样式增强（对比度、背景与边框）；节点与道路 tooltip 展示关键业务字段（建筑类型中文映射、起终点、距离等）。

#### 1.2.3 OSM 数据生成与内存优化

- 优化 `scripts/osm_seed.py` 中道路虚拟节点（`type=virtual_node`）输出：移除非必要的 `description` / `createTime` / `updateTime` 字段，保留定位与关联所需字段，降低运行时内存占用与序列化成本。
- 对生成结果执行脚本化校验，确认虚拟节点规模与字段约束符合预期，保证与现有导入 / 合并流程兼容。

#### 1.2.4 工程化与协作记录

- 持续通过 `docs/AI/HANDOFF.md` 记录跨会话实现结论、验证方式与变更要点，保证与《项目计划书》《技术设计文档》中接口与约束描述可追溯对齐。
- 本周仍以 `mvn` 编译与 `npm run build` 作为基础质量门槛，对路线相关前后端改动进行回归。
